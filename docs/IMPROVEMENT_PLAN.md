# Vibe Coding Simulator 可玩性改进规划

## 一、核心问题诊断

两轮独立分析得出一致结论：游戏的完成度没有问题，文案幽默感有基础，但 **核心决策循环偏弱**，导致"不好玩"。

具体表现为五个症状：

| # | 症状 | 根因 |
|---|------|------|
| 1 | 选择没有策略感 | Accept/Reject 二选一太浅，结果靠随机权重，玩家无法"学会更好的打法" |
| 2 | 事件像孤立笑话 | 事件之间无因果链，前面的选择不影响后面的处境，没有"滚雪球"感 |
| 3 | 项目换皮不换玩法 | 6 个项目只改文案/难度标签，局内循环完全相同，缺乏重开动力 |
| 4 | 过程像平推 | 被动进度 0.25%/s 让不做选择也能赢，缺少阶段目标和节奏曲线 |
| 5 | 后果太抽象 | 数值扣减不可感知，坏决策没有形成可见的局势恶化 |

**一句话总结：** 玩家打完一局，说不出"这局我是怎么打的"。乐趣停留在第一次看段子，缺少决策深度和叙事张力带来的复玩性。

---

## 二、设计目标

改进后的游戏应该满足四层乐趣模型：

```
第 1 层 ✅ 已有  ｜文案好笑，30 秒内吸引人
第 2 层 🚧 核心  ｜每次选择有真实 trade-off，玩家会犹豫
第 3 层 🚧 核心  ｜不同项目/路线能打出不同风格的 run
第 4 层 🚧 核心  ｜玩家会复盘，觉得"这局赢输是我自己打出来的"
```

**检验标准：** 每个改动都要能回答 "是" —— "这个改动会不会让玩家更早形成策略，并且让这一局更像他自己的故事？"

---

## 三、改进方案（按优先级排序）

### P0 — 核心体验重塑（不做这些，其他都是锦上添花）

#### 1. 多选项决策系统

**现状：** 每个事件只有 Accept / Reject。
**目标：** 每个事件 3~4 个选项，每个选项代表不同的策略取向。

**设计思路：**

```
事件：AI 要求安装 847 个 npm 包

选项 A「全盘接受」
  → 进度 +8%, 成本 +$5, 70% 正常 / 20% 依赖冲突 / 10% 灾难
  → 策略定位：速度流

选项 B「只装核心 20 个」
  → 进度 +3%, 成本 +$1, 85% 正常 / 15% 慢一点
  → 策略定位：工匠流

选项 C「让 AI 自己筛选」
  → 进度 +5%, 成本 +$3, 60% 惊喜 / 25% 一般 / 15% AI 选了废包
  → 策略定位：委托流（新风险维度：AI 可靠度）

选项 D「拒绝，手写依赖管理」
  → 进度 +1%, 质量 +5, 成本 +$0.5
  → 策略定位：抠门/洁癖流
```

**关键原则：**
- 选项之间不存在"明显最优解"，而是不同维度的 trade-off
- 每个选项隐含一种玩法风格，让玩家逐渐形成"这局我走什么路线"的意识
- 可以给选项附带部分可见信息（风险等级、预估收益范围），让玩家能做有根据的判断，而非纯赌博

**数据结构变更（示意）：**

```typescript
// 现有
interface GameEvent {
  acceptOutcomes: Outcome[]
  rejectOutcomes: Outcome[]
}

// 改为
interface GameEvent {
  choices: EventChoice[]  // 3~4 个选项
}

interface EventChoice {
  id: string
  label: string            // "全盘接受"
  description: string      // 简短说明 trade-off
  style: PlayStyle         // 'speed' | 'craft' | 'budget' | 'gambler'
  riskLevel: 'low' | 'medium' | 'high'
  estimatedProgress: string // "+3%~+8%" 模糊范围，给玩家判断依据
  outcomes: WeightedOutcome[]
}
```

#### 2. 阶段里程碑系统

**现状：** progress 是 0→100% 的抽象数字，过程像平推。
**目标：** 每局拆成 4~6 个具名里程碑，每个阶段有明确的叙事主题和阶段 Boss 事件。

**设计思路：**

```
阶段 1 (0%~20%)   「项目启动」 — 搭环境、选技术栈
阶段 2 (20%~45%)  「核心开发」 — 实现主要功能，技术债开始积累
阶段 3 (45%~65%)  「需求变更」 — 老板/甲方突然改需求（阶段 Boss）
阶段 4 (65%~85%)  「测试上线」 — bug 集中爆发，部署风险高
阶段 5 (85%~100%) 「最终交付」 — demo day / 上线首日（终局 Boss）
```

**每个阶段转换时：**
- 弹出阶段总结（"你用了 47 秒完成核心开发，累积 3 个 bug"）
- 触发一个阶段专属 Boss 事件（不可跳过，必须面对）
- 根据当前局势状态调整后续事件池的权重

**Boss 事件示例：**

```
阶段 3 Boss：「产品经理紧急会议」

  "距离 demo day 还有 3 天，老板要求加 AI 语音功能。"

  选项 A「硬着头皮上」→ 进度 +15% 但质量 -20, 健康 -15
  选项 B「谈判砍需求」→ 进度 +5%, 质量不变，触发后续"老板不满"状态
  选项 C「通宵加班」  → 进度 +10%, 健康 -25, 后续事件概率加速
  选项 D「提议下个版本再做」→ 进度 +0%, 获得"技术信用"buff
```

#### 3. 连锁后果 & 持续状态系统

**现状：** 每个事件的后果是即时数值加减，互不关联。
**目标：** 坏决策不只扣分，而是在局内施加持续状态，改变后续事件的概率和选项。

**持续状态设计：**

| 状态名 | 触发条件 | 效果 | 消除方式 |
|--------|---------|------|---------|
| `技术债累积` | bug > 5 或连续 3 次选速度流 | 后续重构类事件风险 +20% | 成功完成一次重构事件 |
| `脆弱架构` | systemHealth < 50 | 部署类事件灾难概率翻倍 | systemHealth 回到 70 以上 |
| `预算警报` | apiCost > $100 | 强制插入"老板削预算"事件，高成本选项被锁定 | 连续 3 次选低成本选项 |
| `团队过度自信` | 连续 5 次无负面结果 | AI 推荐更激进方案，高风险事件出现率提升 | 经历一次灾难事件 |
| `士气低落` | 连续遭遇 2 次灾难 | 所有选项的正面结果概率 -10% | 成功完成一个阶段 |
| `技术信用` | 选择保守/高质量选项 | 下一个 Boss 事件解锁额外选项 | 使用一次后消耗 |

**因果链示例：**

```
玩家 Accept「装 847 个 npm 包」→ 触发"依赖地狱"
  ↓
获得状态「脆弱架构」
  ↓
3 个事件后出现「node_modules 磁盘爆满」（因果事件，只有之前选了才出现）
  ↓
如果再处理不好 → 进入「连环崩溃」状态
  ↓
阶段 Boss 难度显著提升
```

**数据结构变更（示意）：**

```typescript
interface GameState {
  // ...现有字段
  activeEffects: ActiveEffect[]  // 当前生效的持续状态
  eventHistory: string[]         // 已触发事件 ID 记录（用于因果链查询）
}

interface ActiveEffect {
  id: string
  name: string           // "技术债累积"
  icon: string           // "🔧"
  description: string
  modifiers: Modifier[]  // 对事件概率/选项的具体修改
  clearCondition: string // 消除条件描述
}

interface GameEvent {
  // ...现有字段
  prerequisites?: string[]    // 前置事件 ID，只有触发过才出现
  requiredEffects?: string[]  // 需要某个持续状态才出现
}
```

### P1 — 差异化与深度（P0 做完后优先）

#### 4. 项目专属机制

**现状：** 6 个项目共享同一个事件池，只有文案差异。
**目标：** 每个项目拥有专属事件、专属风险倾向、以及独特机制。

**设计思路：**

| 项目 | 核心机制差异 | 专属风险 | 推荐流派 |
|------|-------------|---------|---------|
| 区块链 TODO | 引入"Gas 费"作为额外成本维度；部署需要"上链确认"等待时间 | 合规审查事件、智能合约漏洞 | 赌徒流 |
| AI 计算器 | API 成本波动更大；引入"模型幻觉"随机干扰 | 模型打架事件、成本失控 | 抠门流 |
| 宠物社交 | 需求变更频率翻倍；用户反馈事件更多 | 舆情危机、功能膨胀 | 速度流 |
| 植物交友 | IoT 硬件兼容性事件；环境因素随机干扰 | 传感器故障、数据噪声 | 工匠流 |
| 创业操作系统 | 引入"融资轮"机制，花钱多但阶段性获得大量资源 | 投资人施压、pivot 事件 | 全流派高波动 |
| 元宇宙 Excel | 性能瓶颈事件更多；引入"渲染帧率"额外指标 | VR 眩晕投诉、算力不足 | 工匠流 |

**每个项目还应有：**
- 5~8 个专属事件（在通用事件池之上叠加）
- 专属的阶段里程碑名称（比如区块链 TODO 的阶段 3 是"代币经济审计"，而不是通用的"需求变更"）
- 专属的结局文案和称号

#### 5. 玩法风格（流派 Build）显性化

**现状：** 玩家没有"这局我走什么路线"的意识。
**目标：** 让玩家在前 30 秒就开始形成策略倾向，并在结算时明确反映。

**四种基础流派：**

```
⚡ 速度流 — 优先推进度，容忍 bug 和低质量
💎 工匠流 — 优先质量和稳定性，节奏慢但扎实
💰 抠门流 — 严格控成本，追求效率评分
🎰 赌徒流 — 专吃高风险高收益，追求爆发式通关
```

**实现方式：**
- 每个选项标记所属流派（可以不显示给玩家，但内部统计）
- 当玩家连续 3 次选择同一流派的选项后，触发流派确认提示："你似乎在走速度流路线"
- 流派 buff：确认后该流派的选项获得小幅加成（比如速度流的进度 +10%）
- 结算时展示"你的游戏风格：⚡ 速度流"以及流派专属称号
- 流派不锁定，中途可以变，但专注一种流派会有小奖励

#### 6. 决策界面多样化

**现状：** 所有事件都是同一种"权限弹窗"形式。
**目标：** 引入 3~4 种不同的决策界面，打破节奏单调感。

| 界面类型 | 触发场景 | 视觉差异 | 玩家体验 |
|---------|---------|---------|---------|
| 权限弹窗 | AI 请求执行操作 | 现有设计（保留） | 常规决策 |
| 紧急警报 | 线上事故 / 系统告警 | 红色全屏闪烁，倒计时 10 秒 | 紧张，限时抉择 |
| 会议室 | 老板/甲方/投资人施压 | 对话气泡式 UI，NPC 头像 | 社交博弈感 |
| 技术评审 | 代码质量/架构决策 | 代码 diff 视图 | "专业"感，仔细权衡 |

### P2 — 复玩动力（前两层做好后添加）

#### 7. 轻度 Meta 进度

**原则：** 不做重度养成，只做"给再来一局一个明确动机"。

- **项目解锁**：初始只开放 2 个 Easy 项目，完成后解锁 Normal，再解锁 Hard
- **事故图鉴**：收集所有遇到过的事件和结局（"你还有 12 个灾难结局没见过"）
- **称号册**：收集所有称号（"10x Vibe Engineer"、"VC-Funded Hello World" 等）
- **人格解锁**：完成特定挑战后解锁起始 buff，如：
  - 「预算狂魔」：初始 apiCost 上限 +50%（允许更高花费）
  - 「重构洁癖」：初始 codeQuality 80（更高起点）
  - 「增长黑客」：进度增长 +15%，但 bug 率 +20%

**存储方式：** localStorage，不需要后端。

#### 8. 成就系统

**设计少量高质量成就，而非大量水成就：**

| 成就 | 条件 | 难度 |
|------|------|------|
| 「YOLO 工程师」 | 全程只选最高风险选项并通关 | Hard |
| 「零 Bug 传说」 | 0 bug 通关任意项目 | Hard |
| 「一分钱掰两半」 | apiCost < $10 通关 | Medium |
| 「速通大师」 | 120 秒内通关 | Medium |
| 「完美主义者」 | S 评级通关 | Hard |
| 「灾难收藏家」 | 在一局中触发 5 种不同灾难 | Medium |
| 「六边形战士」 | 每个项目都至少 B 评级通关 | Hard |
| 「故事大王」 | 触发一条完整的因果链（从起因到最终后果）| Easy |

---

### 不建议优先做的方向

以下方向看似丰富，但不解决核心问题，应放在 P0~P2 之后：

| 方向 | 为什么不急 |
|------|-----------|
| 继续扩写搞笑事件文案 | 内容堆积不等于玩法增强，现有 32 个事件改为多选项后工作量已经很大 |
| 增加更多项目皮肤 | 6 个项目做出真正差异化比加到 10 个但都一样更重要 |
| 优化结算页面动画 | 现有动画已经够用，核心问题不在这 |
| 增加更多数值维度 | 数值多不等于决策深度，关键是让现有数值之间产生关联和博弈 |
| 小游戏/QTE 操作 | 开发成本高，且与"氛围编程模拟"的核心体验定位有冲突 |

## 四、实施路线图

### 第一阶段：核心循环重做（最高优先级）

```
目标：让单局体验从"看段子"变成"做权衡"

 ┌─────────────────────────────────────────────────────┐
 │  1.1  多选项决策系统                                   │
 │       - 改 EventChoice 数据结构                        │
 │       - 重写 PermissionModal 支持 3~4 选项             │
 │       - 改造前 10 个高频事件为多选项（验证手感）           │
 │                                                       │
 │  1.2  阶段里程碑                                       │
 │       - 定义 5 个阶段及其主题                            │
 │       - 阶段转场 UI（总结卡片）                          │
 │       - 实现 3 个 Boss 事件（阶段 3 / 4 / 5 各一个）     │
 │                                                       │
 │  1.3  连锁后果系统                                      │
 │       - ActiveEffect 数据结构和状态管理                   │
 │       - 实现 4 个核心持续状态（技术债/脆弱架构/预算警报/过度自信）│
 │       - 实现 2~3 条因果链                               │
 │                                                       │
 │  1.4  调整被动进度                                      │
 │       - 降低 IDLE_PROGRESS_PER_SECOND（0.25 → 0.10）   │
 │       - 让进度主要由玩家决策驱动                          │
 └─────────────────────────────────────────────────────┘
```

**验收标准：** 内测时玩家能说出 "这局我走的是 XX 路线，因为前面选了 XX 所以后面遇到了 XX"。

### 第二阶段：差异化

```
目标：让"再开一局"有明确理由

 ┌─────────────────────────────────────────────────────┐
 │  2.1  项目专属事件和机制                                │
 │       - 每个项目设计 5~8 个专属事件                      │
 │       - 每个项目设计专属里程碑名称                        │
 │       - 实现项目专属风险倾向                              │
 │                                                       │
 │  2.2  流派系统显性化                                    │
 │       - 选项流派标记 + 统计                              │
 │       - 流派确认提示 + 小 buff                           │
 │       - 结算页展示玩法风格                               │
 │                                                       │
 │  2.3  决策界面多样化                                    │
 │       - 紧急警报界面（限时 + 红色主题）                    │
 │       - 会议室界面（NPC 对话式）                          │
 │       - 技术评审界面（代码 diff 式）                      │
 │                                                       │
 │  2.4  改造剩余事件                                      │
 │       - 将剩余 22 个事件全部改为多选项                     │

 └─────────────────────────────────────────────────────┘
```

**验收标准：** 玩家主动想试不同项目，并且发现打法确实不同。

### 第三阶段：复玩闭环

```
目标：给予长期动力

 ┌─────────────────────────────────────────────────────┐
 │  3.1  项目解锁 + 事故图鉴 + 称号册                      │

```

**验收标准：** 玩家在通关 3 局后仍有明确的"再来一局"动机。

## 五、预期效果

```
改进前：
  "哈哈这个文案好搞笑" → 玩 2 局 → "就这样吧"

改进后：
  "哈哈这个文案好搞笑"                          ← 第 1 层（已有）
  → "这个选项好纠结，选 A 快但可能炸"             ← 第 2 层（P0 多选项）
  → "上局走速度流翻车了，这次试试工匠流"           ← 第 3 层（P1 流派 + 项目差异）
  → "那局输在阶段 3 Boss，因为前面技术债太多了"     ← 第 4 层（P0 连锁后果 + 里程碑）
  → "还有 3 个成就没解锁，Hard 项目还没试"         ← 复玩动力（P2）
```
